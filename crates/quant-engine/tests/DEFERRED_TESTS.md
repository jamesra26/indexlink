## 1. 高估但趋势强

- **场景**：CAPE 高、ERP 低（基本面贵），但近期趋势很好（200 日均线之上、RSI/动量强）。
- **预期**：基本面层仍判定「贵」；趋势层只能在**节奏**上微调（20% 权重），
  **不能推翻 70% 主锚**——综合得分仍应落在偏贵区间。
- **依赖**：
  - `quant-engine` 趋势层从存根 `evaluate_trend_stub()` 升级为接受真实输入的实现；
  - `decision` crate 提供 70/20/10 合成函数。
- **落地位置（建议）**：`crates/decision/tests/` 的合成场景测试。
- **断言要点**：
  - 趋势分给出最强烈的「便宜/加码」信号时，综合得分相对纯基本面得分的下移量
    **不超过趋势层权重所能贡献的上限**（量化「主锚不被推翻」）。
  - 最终动作不得因趋势强而从 `Underweight`/`Skip` 跳到 `Overweight`。

## 2. 低估但急跌（不接飞刀）

- **场景**：CAPE 低、ERP 高（基本面便宜），但趋势/波动恶化（急跌、VIX 飙升）。
- **预期**：不应激进加码，体现「不接飞刀」——趋势层给出负向修正使加码更保守。
- **依赖**：
  - 趋势层需接入波动率 / 急跌信号（VIX、回撤速度等），当前无此输入字段；
  - `decision` crate 的合成与 `clamp` 映射。
- **落地位置（建议）**：`crates/decision/tests/`。
- **断言要点**：
  - 同样的便宜基本面下，趋势恶化版本的最终倍率应**显著低于**趋势中性版本；
  - 即便基本面极便宜，急跌情形下倍率不应触顶到最大 Overweight。

## 3. 审计回放（确定性）

- **场景**：同一输入快照重复评估，必须得到逐位相同的输出。
- **预期**：可复现性是审计前提（见 `readme.md`「金融可靠性三件套：审计」）。
- **现状说明**：当前 `quant-engine` 全为纯函数、零 IO、无随机/全局状态，
  确定性**由构造保证**；此刻写一条 `assert_eq!(run(x), run(x))` 价值有限，
  仅作防回归守卫（防止未来误引入 `HashMap` 迭代序、并行、时间戳等不确定来源）。
- **真正的回放能力**依赖**输入快照的保存与反序列化**（`readme.md` 阶段 4 的
  feature-gated `serde`），且反序列化必须复用 newtype 不变量校验。
- **依赖**：`core-domain` / `quant-engine` 的 `serde` 支持；`decision` 的 `Decision`（含输入快照）。
- **落地位置（建议）**：
  - 轻量回归守卫：可在引入并行/缓存等优化的同一改动里补到对应 crate 的测试；
  - 完整回放：`serde` 落地后，在快照「序列化 → 反序列化 → 重新评估 → 结果一致」链路上测试。
- **断言要点**：
  - 反序列化后的快照重新评估，得分与动作与原始完全一致；
  - 越界/NaN 等非法快照在反序列化阶段即被不变量校验拒绝，不得静默通过。
